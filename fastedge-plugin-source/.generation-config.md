# Generation Config — AssemblyScript Proxy-Wasm SDK

> **What this file is**: Structured instructions for generating `docs/` from source code.
> **How to use it**: Run `./fastedge-plugin-source/generate-docs.sh` which reads this file and invokes Claude per doc file.
> **Who reads this**: The generation script + Claude CLI. The plugin pipeline does NOT read this file.

---

## Global Rules

### Audience

These docs are for **consumers** of `@gcoredev/proxy-wasm-sdk-as` — developers building CDN filter applications (proxy-wasm plugins) that compile to WebAssembly and run on the FastEdge CDN platform. They are NOT for contributors developing this SDK.

This distinction drives what to include and exclude:

- **Include**: anything a consumer needs to write a CDN app (public API, lifecycle hooks, host APIs, build configuration)
- **Exclude**: internal implementation details, contributor workflows, raw host function declarations, memory management internals
- **Exclude**: any proxy-wasm feature not supported on the FastEdge platform (see Platform Scope below)
- **Test**: "Would someone adding `@gcoredev/proxy-wasm-sdk-as` to their project need to know this?" If no, leave it out.

### Platform Scope — CRITICAL

This SDK is a fork of the Kong proxy-wasm AssemblyScript SDK. It inherits many types, classes, and enums from the upstream proxy-wasm specification that are **not implemented** on the FastEdge CDN platform. These MUST be completely omitted from the documentation — do not mention them, do not mark them as "unsupported", just leave them out entirely.

**If a feature is not documented in these docs, it is not supported on FastEdge.** This documentation is the authoritative source of truth for what is available. AI agents and developers should not rely on general proxy-wasm knowledge from training data or external documentation — only what is explicitly documented here.

### Style

- Technical prose only — no marketing language, no superlatives, no "easily" or "simply"
- Use TypeScript-style code signatures for all types, functions, and methods (this is AssemblyScript — a strict subset of TypeScript that compiles to WebAssembly)
- Use fenced code blocks for all examples (language-tagged as `typescript`)
- Use tables for structured data (methods, enum values, config fields)
- Tables must use padded columns aligned to the widest cell in each column. Pad every cell with spaces so columns line up visually in raw markdown. The separator row dashes must match each column's padded width.
  Good: `| Method                | Return Type          | Description |`
  Bad: `| Method | Return Type | Description |`
  Good: `| --------------------- | -------------------- | ----------- |`
  Bad: `|---|---|---|`
- Every code example must be self-contained and representative of real usage

### Structure

- Every doc file starts with a level-1 heading and a one-line description
- Use level-2 headings for major sections, level-3 for subsections
- No table of contents — keep files navigable by heading structure alone
- End each file with a "See Also" section linking ONLY to other files in `docs/`

### Platform Constraints (apply to all files)

- **Logging**: Use `log(LogLevelValues.info, "message")` from the SDK. This routes through the proxy-wasm host to stdout. `console.log` is NOT available in the WebAssembly environment. State this explicitly in any section discussing logging.
- **No try/catch**: AssemblyScript does not support `try/catch` in most contexts. Error handling is done via return values and null checks, not exceptions.
- **No closures over mutable state**: AssemblyScript prohibits closures that capture mutable variables. The `registerRootContext` factory callback is the notable exception (it captures nothing mutable).
- **Explicit numeric types**: Use `u32`, `i32`, `usize`, `f64`, `u64`, `bool` — NOT `number` or `boolean`. These are AssemblyScript-specific types, not standard TypeScript.

### Import Paths — CRITICAL

There are exactly two import paths consumers use:

1. **Core SDK**: `@gcoredev/proxy-wasm-sdk-as/assembly` — RootContext, Context, enums, stream_context, log, registerRootContext, header manipulation functions, buffer functions, property functions
2. **FastEdge extensions**: `@gcoredev/proxy-wasm-sdk-as/assembly/fastedge` — getEnv, getSecret, getSecretEffectiveAt, KvStore, getCurrentTime, setLogLevel

Additionally, every plugin must re-export proxy entry points:

```typescript
export * from "@gcoredev/proxy-wasm-sdk-as/assembly/proxy";
```

CRITICAL: Never import from sub-paths like `assembly/fastedge/dictionary` or `assembly/fastedge/kvStore` directly — always import from the barrel export `assembly/fastedge`. Never import from `assembly/runtime` directly — use `assembly` (the barrel export in `assembly/index.ts`).

### Exclusions (apply to all files)

- No internal implementation details (how `ArrayBufferReference` works, how `malloc.ts` manages memory, how `exports.ts` dispatches to contexts)
- No source code file paths or line numbers
- No version history or changelog entries
- No references to `context/` or `CLAUDE.md` (those are internal developer docs)
- No raw host function declarations from `imports.ts` (`@external("env", ...)` functions)
- No `changetype<usize>()` pointer casting details (internal only)

### Accuracy

- Never hardcode the SDK version — read it from `package.json`
- **Dependency versions must come from source files, never invented.** Read the `assemblyscript` version from `package.json`. Read the `@assemblyscript/wasi-shim` version from `package.json`. Never use `"*"` or unversioned dependencies.
- All type signatures must match actual source code declarations
- AssemblyScript types are NOT standard TypeScript: `u32` (not `number`), `usize` (not `number`), `bool` (not `boolean`), `f64` (not `number`), `u64` (not `number`/`bigint`)

### Coverage Rule

- Every public export, class, method, function, and enum must appear in exactly one doc file
- If something is referenced in multiple files, one file is authoritative (has the full definition) and others link to it

### Scope

Complete API reference for this CDN-only SDK. Everything a developer needs to write a FastEdge CDN application (proxy-wasm filter) in AssemblyScript.

**This file does NOT cover**: Step-by-step getting started (→ quickstart.md). This file is the exhaustive API reference.

### Required Content

**Proxy-Wasm Lifecycle — document thoroughly as this is the most important concept:**

1. **Entry point pattern** — every plugin must:
   - `export * from "@gcoredev/proxy-wasm-sdk-as/assembly/proxy"` (exposes wasm entry points to the host)
   - Call `registerRootContext(factory, name)` at module scope
   - Have `asconfig.json` with `"use": "abort=abort_proc_exit"`
2. **RootContext** — created once per VM instance
   - `createContext(context_id: u32): Context` — factory method called per-request
   - Optional overrides: `onStart(config_size: u32): bool`, `onDone(): bool`
   - `httpCall(uri: string, headers: Headers, body: ArrayBuffer, trailers: Headers, timeout: u32): u32` — make an outbound HTTP call (public hosts only; certain headers like `host`, `content-length`, `transfer-encoding` are auto-filtered)
3. **Context** — instantiated per lifecycle hook invocation (NOT per-request — see Hook State Isolation above). This is where request/response processing happens.
   - Constructor pattern: `constructor(context_id: u32, root_context: RootContext)`
   - Lifecycle hooks (document each with exact signature and return type):
     - `onRequestHeaders(headers: u32, end_of_stream: bool): FilterHeadersStatusValues`
     - `onRequestBody(body_buffer_length: usize, end_of_stream: bool): FilterDataStatusValues`
     - `onResponseHeaders(a: u32, end_of_stream: bool): FilterHeadersStatusValues`
     - `onResponseBody(body_buffer_length: usize, end_of_stream: bool): FilterDataStatusValues`
     - `onLog(): void`

**CRITICAL — Hook State Isolation:**

On the FastEdge CDN platform, **a Context instance only exists for the duration of a single lifecycle hook invocation**. It does NOT persist across the request. Different hooks may run on entirely different servers — `onRequestHeaders` runs in nginx, while `onRequestBody`, `onResponseHeaders`, `onResponseBody`, run in core-proxy.

This means:

- Instance fields on Context subclasses are NOT available in subsequent hooks
- `this.root_context` references are NOT available across hooks
- The Context constructor runs fresh for each hook invocation

To pass data between hooks (e.g., from `onRequestHeaders` to `onResponseHeaders`), use `set_property` / `get_property`:

```typescript
// In onRequestHeaders:
set_property("my_custom_key", String.UTF8.encode("my_value"));

// In onResponseHeaders:
const value = get_property("my_custom_key"); // retrieves the value set earlier
```

Do NOT document or suggest:

- "Context is created once per HTTP request" — WRONG, it is created per hook invocation
- "Store state as instance fields" — WRONG, fields do not persist
- "Access shared state through this.root_context" — WRONG, root_context is not available across hooks
- Any pattern that implies the same Context object is reused across hooks

**Return Value Enums — document every value and what it means:**

- `FilterHeadersStatusValues`: `Continue`, `StopIteration`, `ContinueAndEndStream`, `StopAllIterationAndBuffer`, `StopAllIterationAndWatermark`
- `FilterDataStatusValues`: `Continue`, `StopIterationAndBuffer`, `StopIterationAndWatermark`, `StopIterationNoBuffer`
- `LogLevelValues`: `trace`, `debug`, `info`, `warn`, `error`, `critical`
- `WasmResultValues`: `Ok`, `NotFound`, `BadArgument`, `SerializationFailure`, `ParseFailure`, `BadExpression`, `InvalidMemoryAccess`, `Empty`, `CasMismatch`, `ResultNotFound`, `InternalFailure`, `BrokenConnection`, `Unimplemented`
- `BufferTypeValues`: `HttpRequestBody`, `HttpResponseBody`, `HttpCallResponseBody` (only these three are usable on FastEdge)

**Request/Response Manipulation via stream_context:**

- `stream_context.headers.request` and `stream_context.headers.response` — both provide:
  - `get(name: string): string` — get a header value
  - `add(name: string, value: string): void` — add a header
  - `replace(name: string, value: string): void` — replace a header value
  - `remove(name: string): void` — remove a header. **Known limitation**: on the FastEdge CDN (nginx-based), `remove` sets the header value to an empty string rather than fully removing it. Keep this in mind when checking for header presence.
  - `get_headers(): Headers` — get all headers as `HeaderPair[]`
  - `set_headers(headers: Headers): void` — replace all headers

CRITICAL: Response headers are only available during response-phase hooks (`onResponseHeaders`, `onResponseBody`). Accessing them during request-phase hooks will panic.

**Body and Buffer Functions:**

- `get_buffer_bytes(type: BufferTypeValues, start: u32, length: u32): ArrayBuffer` — read body/buffer content
- `set_buffer_bytes(type: BufferTypeValues, start: u32, length: u32, value: ArrayBuffer): WasmResultValues` — write body/buffer content
- Common pattern: use `BufferTypeValues.HttpRequestBody` in `onRequestBody` and `BufferTypeValues.HttpResponseBody` in `onResponseBody`

**Property Functions:**

- `get_property(path: string): ArrayBuffer | null` — read a runtime property
- `set_property(path: string, value: ArrayBuffer): WasmResultValues` — set a custom runtime property

Available properties on the FastEdge platform (document ALL of these):

| Property               | Encoding              | Description                                                                     |
| ---------------------- | --------------------- | ------------------------------------------------------------------------------- |
| `request.path`         | UTF-8 string          | URL path                                                                        |
| `request.query`        | UTF-8 string          | Query string                                                                    |
| `request.url`          | UTF-8 string          | Full request URL                                                                |
| `request.host`         | UTF-8 string          | Domain (may have `shield_` prefix on edge shield nodes)                         |
| `request.scheme`       | UTF-8 string          | HTTP scheme (from X-Forwarded-Proto)                                            |
| `request.extension`    | UTF-8 string          | File extension                                                                  |
| `request.x_real_ip`    | UTF-8 string          | Client IP address                                                               |
| `request.country`      | UTF-8 string          | 2-letter ISO country code (geo-IP)                                              |
| `request.country.name` | UTF-8 string          | Full country name                                                               |
| `request.city`         | UTF-8 string          | City name                                                                       |
| `request.region`       | UTF-8 string          | Region/state                                                                    |
| `request.continent`    | UTF-8 string          | Continent                                                                       |
| `request.asn`          | UTF-8 string          | Autonomous System Number                                                        |
| `request.geo.lat`      | UTF-8 string          | Latitude                                                                        |
| `request.geo.long`     | UTF-8 string          | Longitude                                                                       |
| `response.status`      | 2-byte big-endian u16 | Response status code (**binary, NOT a string** — decode with byte manipulation) |

CRITICAL: `response.status` is binary, not a UTF-8 string. All other properties are UTF-8 strings decoded via `String.UTF8.decode()`. Geo properties are lazily computed from the client IP address.

**Response Generation:**

- `send_http_response(status: u32, status_description: string, body: ArrayBuffer, headers: Headers): void` — send a response directly to the client, bypassing origin
- `send_local_response(status: u32, status_description: string, body: ArrayBuffer, headers: Headers, grpc_status: i32): void` — lower-level variant (prefer `send_http_response` which wraps this with grpc_status defaulted to 0)

**Logging:**

- `log(level: LogLevelValues, message: string): void` — log a message through the proxy-wasm host

**Registration:**

- `registerRootContext(factory: (context_id: u32) => RootContext, name: string): void` — register the root context factory

**FastEdge Host APIs (from `assembly/fastedge`):**

Document each with exact signature and import path:

- `getEnv(name: string): string` — read an environment variable (set at deployment time via the platform)
- `getSecret(name: string): string` — read a secret value
- `getSecretEffectiveAt(name: string, effectiveAt: u32): string` — read a secret with time-based versioning for rotation
- `KvStore` class:
  - `static open(storeName: string): KvStore` — open a named KV store
  - `get(key: string): ArrayBuffer | null` — read a value (null if not found)
  - `scan(pattern: string): string[]` — glob-style key scanning
  - `zrangeByScore(key: string, min: f64, max: f64): ValueScoreTuple[]` — sorted set range query
  - `zscan(key: string, pattern: string): ValueScoreTuple[]` — sorted set pattern scan
  - `bfExists(key: string, item: string): bool` — bloom filter membership test
- `ValueScoreTuple` type — used by `zrangeByScore` and `zscan` results
- `getCurrentTime(): u64` — current time in milliseconds
- `setLogLevel(level: LogLevelValues): void` — set minimum log level (defaults to `info`)

**Deprecated Functions — mark clearly:**

- `getEnvVar(name: string): string` — DEPRECATED, use `getEnv` instead
- `getSecretVar(name: string): string` — DEPRECATED, use `getSecret` instead
- `getSecretVarEffectiveAt(name: string, effectiveAt: u32): string` — DEPRECATED, use `getSecretEffectiveAt` instead

### CRITICAL Accuracy Notes

- `KvStore.get(key)` returns `ArrayBuffer | null` — NOT `string`. The consumer must decode the buffer.
- `KvStore.zrangeByScore` and `KvStore.zscan` return `ValueScoreTuple[]` — document the tuple structure.
- `getSecret` returns `string` — NOT `ArrayBuffer`. The host handles decoding.
- `get_property` returns `ArrayBuffer | null` — most properties are UTF-8 strings, but `response.status` is a 2-byte big-endian u16.
- Lifecycle hook parameter names: `onRequestHeaders` takes `headers: u32` (count), `onRequestBody` takes `body_buffer_length: usize`. Read these from source — do NOT invent parameter names.
- `setLogLevel` is exported from `assembly/fastedge`, NOT from `assembly`. This is a common error.

### DO NOT document

**Internal implementation details:**

- `ArrayBufferReference` class (internal memory management)
- `malloc` / `free` functions (host-side memory allocation)
- `globalArrayBufferReference` (internal singleton)
- `proxy_on_*` dispatch functions in `exports.ts` (internal wasm entry points)
- Raw `@external("env", ...)` host function declarations from `imports.ts`
- `listParser.ts` deserializer internals (consumers never use this directly)
- `changetype<usize>()` pointer casting patterns

**Internal SDK classes and methods not needed by consumers:**

- `BaseContext` class — internal base for RootContext/Context. Developers never use `context_id`, `continueRequest()`, `onDone()`, or `onDelete()` directly.
- `onConfigure(configuration_size: u32)` on RootContext — not called by the FastEdge server, omit entirely.

**Unsupported proxy-wasm features (inherited from Kong SDK, NOT available on FastEdge — omit entirely):**

- `GrpcStatusValues` enum and all gRPC-related types
- `MetricTypeValues` enum and metric classes: `Counter`, `Gauge`, `Histogram`
- `GetSharedData` class and all shared data/queue functions
- `onTick()` hook on RootContext (tick timers are not supported — `proxy_set_tick_period_milliseconds` returns Unimplemented)
- Connection hooks: `onNewConnection`, `onDownstreamData`, `onUpstreamData`, `onDownstreamConnectionClose`, `onUpstreamConnectionClose`
- `FilterStatusValues`, `FilterMetadataStatusValues`, `FilterTrailersStatusValues` enums (metadata and trailer filtering not supported)
- `proxy_clear_route_cache`, `proxy_call_foreign_function`
- `BufferTypeValues` values beyond `HttpRequestBody`, `HttpResponseBody`, `HttpCallResponseBody` (e.g., `DownstreamData`, `UpstreamData`, `GrpcReceiveBuffer` are not usable)
